1. アップロードログ信頼性 + iOS Safari 無音ハング対策 (v2.8.2)

   **課題**

   - iOS Safari 18.7 / iPhone で 577MB .mov アップロードが無音で失敗する現象が発生していた（`bunny-create success` で全ログが途切れ、`tus-upload start` すら記録されない）
   - 欠陥A: `flush()` 成功時に無条件で pending 除外していたため、flush 後に追加されたエントリは離脱時に `retryPendingFlush` の対象外となり永遠に送信されなかった
   - 欠陥B: `bunny-create` から最初の `tus-progress` までの間が診断ログゼロで、Wake Lock 取得・`new tus.Upload`・`findPreviousUploads`・`upload.start()` のどこで固まったか特定不能だった
   - 欠陥B': `chunkSize` 未指定のため iOS Safari で大ファイルの File.slice() 遅延評価が数十秒〜数分ブロックしていた
   - 欠陥C: `onPhaseChange('uploading')` 表示後、最初の progress が来るまで UI が沈黙しユーザが「止まった」と誤認していた

   **変更**

   - `upload-logger.ts`: `flush({ final })` に変更し、final 時のみ pending 除外を評価する watermark 方式に修正。`recordEntry` ヘルパーで全エントリ追加時に `markPendingFlush` + `scheduleAutoFlush` を実行。flush を直列化し待機 caller の final 要求を OR で集約するパターンに変更。`_doFlush` は送信不要時でも final なら `removePendingFlag` を評価。`logInfo(phase, event, extra?)` 共通メソッドを追加し `startNetworkMonitor` の直書きを統一。`installUnloadBeacon(getAccessToken)` / `uninstallUnloadBeacon` を追加し pagehide + visibilitychange=hidden で fetch keepalive 送信
   - `upload-beacon.ts`（新規）: `sendLogsOnUnload` 実装。`fetch(..., keepalive: true)` で Supabase REST に upsert、60KB 超過時はスキップして既存ログの partial 上書きを回避
   - `upload-wakelock.ts`: `acquireWakeLock({ timeoutMs })` に変更。`Promise.race` で 2 秒タイムアウト、戻り値を `{ release, acquired, elapsedMs, timedOut }` に拡張。タイムアウト後に遅れて resolve されたケースも `released` フラグで即座に解放
   - `upload-core.ts`: Wake Lock 取得・`new tus.Upload`・`findPreviousUploads`・`upload.start()` 各ステップに info チェックポイントを挿入（`wakelock-acquire-start/done`・`tus-construct-start/done`・`find-previous-start/done/fail`・`upload-start-called`・`first-progress`・`no-progress-timeout`）。`chunkSize: TUS_CHUNK_SIZE (25MB)` を明示指定。`upload.start()` 後 30 秒以内に progress が来なければ `abort` + reject する初動タイムアウト追加。10 秒で `onStageChange('stalled')` 発火。final flush を `await logger.flush({ final: true })` に変更。beacon 用 token を `onAuthStateChange` で追跡し tus-upload 終了時に unsubscribe
   - `upload-constants.ts`: `TUS_CHUNK_SIZE = 25MB`・`TUS_INITIAL_PROGRESS_TIMEOUT_MS = 30000`・`TUS_STALLED_WARN_MS = 10000`・`WAKELOCK_TIMEOUT_MS = 2000` を追加
   - `AuthContext.tsx`: `retryPendingFlush` を `fetchProfile` 内から `useEffect` 直下に移動。初回 `getSession().then` と `onAuthStateChange` の `SIGNED_IN` イベントから呼び出し、Profile 取得の成否・認証エラーに依存しない堅牢化
   - `UploadModal.tsx`/`PendingUploadCard.tsx`: `ItemUploadState`/`UploadState` に `stage: UploadStage` を追加。`onStageChange` を接続し `preparing-video` → 「動画を作成中...」、`preparing-wakelock`/`preparing-tus` → 「アップロード準備中...」、`uploading` (progress=0) → 「アップロードを開始しています...」、`stalled` → `AlertTriangle` + 黄色「通信が遅延しています。そのままお待ちください」を表示

1. アップロードログ信頼性強化 + ネットワーク詳細ログ (v2.7.2)

   **課題**

   - `retryPendingFlush()`が実装済みだが呼び出し元がなく、flush失敗時のログがlocalStorageに閉じ込められたまま消失していた
   - `retryPendingFlush()`で複数pending処理時、ループ内で毎回書き戻すため成功済みIDが復活するstale IDバグがあった
   - tus-js-clientの内部リトライ（最大4回）がログに記録されず不可視だった
   - ネットワーク状態がフェーズ境界のスナップショットのみで、TUS中のオフライン/オンライン遷移が検知不能だった
   - flushがアップロード終了時の1回のみで、これが失敗するとサーバー上に証拠ゼロだった

   **変更**

   - `flush()`をwatermark方式（`lastFlushedCount`）に変更し、早期flush→最終flushの2回呼び出しを可能にした
   - `retryPendingFlush()`の書き戻しをループ後の一括方式に修正し、stale IDバグを解消
   - `UploadStatus`型に`'info'`を追加し、ネットワークイベントをエラー判定に影響しない形で記録
   - `startNetworkMonitor()`メソッドを追加し、TUS中のonline/offlineイベントを`'info'`ステータスで記録
   - `executeUpload()`にpendingFlushリトライ（開始時）と早期flush（bunny-create成功後）を追加
   - TUSの`onShouldRetry`で既定判定（defaultOnShouldRetry）を再現しつつ、リトライを累積カウンタでログ記録
   - `AuthContext`のログイン時に`retryPendingFlush()`を呼び出し、前回失敗分を自動リトライ
   - 管理画面にリトライ回数（黄色バッジ）・ネットワークイベント数（オレンジバッジ）・`'info'`ステータスのラベル/アイコンを追加

   - `isPreparing` stateを追加し、ハッシュ計算中のアップロードボタン無効化で早期送信を防止
   - `fileSelectCounterRef`によるレースガードを追加し、ファイル再選択・クリア・リセット時の非同期結果上書きを防止
