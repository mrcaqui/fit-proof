- 基本的には設定変更前のものはそのままで設定変更による影響を受けない。過去ストレートを達成して、それによりシールドを獲得していたりしたらそのまま
- ただ、何かミスがあった時などのため、遡って適用範囲を操作できる仕組みは欲しい。これが無いと「4月からストレート4回でシールド1つね！」などとユーザと合意していた場合に確実に4/1などに設定変更しないと整合性が合わなくなる。そういったケースの対処用にある程度の柔軟性は持たせたい
- 「ストレート達成」カードと「シールド」カードにある内容はその週の終わりに条件に合致したか判定するために使用する。その時の設定でストレートやシールド獲得と判定できたならそれで確定してOK。週の途中の変更は新しい設定による基準に合わせる。つまり設定変更しても「週の終わりの、その時の設定で判断して計算してOK」ということ
- 「リバイバル」、「連続日数」、「累積記録」カードは表示の有無だけなので適用期間などは無く、即時でOK

計画や現行動作と照らし合わせて技術的な可否や新たな複雑さが生まれる可能性、矛盾などを指摘してほしいです。

- 既存の「適用開始日」カードはゲーミフィケーションの全ての起点であり全体に影響する設定。連続日数のカウント、累積記録のカウント、リバイバルのカウント、ストレート達成、シールド獲得の全てはここに設定された日付以降からカウントとする。これを「ゲーミフィケーション適用開始日」と呼称する
- 「ストレート達成」と「シールド」は設定変更による柔軟性を確保するため、それぞれ個別の適用開始日を設ける。これらをそれぞれ「ストレート達成適用開始日」、「シールド獲得適用開始日」と呼称する
- 設定変更時に設定履歴として保存されているが、これを「ストレート達成適用開始日」、「シールド獲得適用開始日」それぞれの設定履歴管理に分ける。履歴は既存のようにトグルリストに格納してUIの表示の妨げにならないようにし、必要に応じ確認できる形とする
- 設定履歴の適用終了日の記録は廃止し、適用開始と設定変更日時の記録とする
- 設定は最も新しい適用開始日のものが現行であると判断し、適用開始日が同じ日付のものは設定変更日時が最も新しいものを現行とする
- こうすることで適用終了日を用いなくても現行設定の適用開始日=前の設定の適用終了日と判断でき、設定反映の重複が起こらないことが期待される
- ストレート達成もシールド獲得も、その週の判定を行うその時点の設定を適用して該当したかどうかを判断したいが、この判断がどのように行われるのか把握できてないため説明してほしい。決まった時間(日曜日の23:59分など)？ユーザがアプリにアクセスする度にその時の適用開始日などを再計算？
